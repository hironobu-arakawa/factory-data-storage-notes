# Storage is a decision, not a container

このドキュメントは、  
factory-data-storage-notes 全体の「立ち位置」を明確にするための導入です。

---

## なぜ「保存」は設計判断なのか

工場データ基盤では、次の判断が日常的に行われます。

- このデータは保存するか？
- どこに保存するか？
- どの粒度で保存するか？
- 一度保存したら、意味は固定されてしまわないか？

多くの場合、これらは次のように扱われます。

> 「あとで使うかもしれないから、とりあえず保存しておこう」

しかし工場データにおいて  
**保存は可逆ではありません**。

- 保存した瞬間に
  - 粒度
  - 時間解像度
  - 意味の前提
  - 責任の所在  
  が固定されます

これは単なるストレージ選択ではなく、  
**将来の解釈可能性を制限する意思決定**です。

---

## 保存すると「何が確定してしまうか」

保存によって確定するもの：

- 「この値は事実である」という宣言
- 補正・欠損・異常の扱い方
- 後続システムが前提としてよい意味
- 運用が介入できる余地の有無

保存とは、VCD / BOA 的に言えば：

- **Fact をどこで固定するか**
- **Interpretation を後段に残せるか**
- **Responsibility を誰に置くか**

を同時に決める行為です。

---

## このリポジトリが扱う範囲

このリポジトリは、以下を扱います。

- 保存すべき／すべきでない判断軸
- DB / TSDB / Object Storage を分ける理由
- 「後で意味が変わる可能性」をどう残すか
- データ保障（正常・異常・補正）の扱い方

一方で、次のものは扱いません。

- データの形（縦持ち / 横持ち / event など）
- 製品比較やツール選定
- 実装レベルのスキーマ設計

それらは **別のリポジトリの責務**です。

---

## 誰のためのノートか

- 保存判断で毎回迷っている設計者
- 「全部保存」に違和感を覚えている人
- 運用を壊さないデータ基盤を作りたい人
- 判断を AI に委ねる前提を作りたい人

このノートは、  
**正解を教えるためのものではありません**。

判断を「再現可能」にするための痕跡です。
